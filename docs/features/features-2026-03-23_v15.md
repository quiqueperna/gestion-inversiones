## Checklist de cambios

### 1. Data Layer — Prisma + Supabase PostgreSQL
- `prisma/schema.prisma`: provider `sqlite` → `postgresql`
- datasource `url` → `env("POSTGRES_PRISMA_URL")` (pooling puerto 6543)
- datasource `directUrl` → `env("POSTGRES_URL_NON_POOLING")` (directo puerto 5432)
- `src/server/db.ts`: singleton serverless con `datasourceUrl: process.env.POSTGRES_PRISMA_URL`

### 2. Auth Integration — @supabase/ssr
- `src/utils/supabase/server.ts`: cliente Supabase para Server Components / Server Actions
- `src/utils/supabase/client.ts`: cliente Supabase para Client Components (browser)
- `src/middleware.ts`: refresca sesión en cada request, matcher excluye estáticos

### 3. Variables de entorno
- `.env.example`: documentación completa de todas las variables necesarias
- `.env`: actualizado con placeholders para Supabase

### 4. Testing Preservation
- `tsconfig.json`: excluidos `prisma/seed-complete.ts` y `prisma/seed-csv.ts` (seeds legacy con modelos obsoletos)
- Tests unitarios (Vitest): sin cambios, usan in-memory data-loader, no requieren DB

### 5. CI/CD — GitHub Actions
- `.github/workflows/ci.yml`: actualizado con step explícito `prisma generate`
- Variables de entorno mock para CI (con fallback a valores placeholder)
- Secrets de Supabase tomados de `secrets.*` cuando están configurados en el repo

### 6. Server Action de referencia
- `src/server/actions/example-migrated.ts`: ejemplo completo de Server Action con:
  - Validación de sesión Supabase (`supabase.auth.getUser()`)
  - Validación de input con Zod
  - Acceso a datos con Prisma
  - Resultado tipado (`ActionResult<T>`)
