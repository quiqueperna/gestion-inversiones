## Aislamiento de datos por usuario

### Problema resuelto
Usuarios diferentes compartían datos porque el `memoryState` era un singleton global y las tablas DB no tenían campo `userId`.

### 1. Schema Prisma — campo `userId` en todos los modelos
- `Execution`, `TradeUnit`, `CashFlow`: agregado `userId String @default("")` + `@@index([userId])`
- `Account`, `Broker`: cambiado `nombre @unique` → `@@unique([userId, nombre])` para permitir que distintos usuarios tengan cuentas/brokers con el mismo nombre

### 2. data-loader.ts — memoryState per-usuario
- Reemplazado singleton `let memoryState` por `Map<string, MemState>` (`stateMap`)
- Todas las funciones exportadas reciben `userId: string = '_test_'` como último parámetro
- Tests siguen usando el default `'_test_'` sin modificaciones

### 3. Helper getCurrentUserId (`src/server/actions/get-user.ts`)
- Función server-side que obtiene el `user.id` de Supabase SSR autenticado

### 4. Server actions — filtrado completo por userId
Archivos actualizados: `trades.ts`, `transactions.ts`, `dashboard.ts`
- `ensureDataLoaded()` / `ensureLoaded()` retornan `{ state, userId }`
- Todas las queries DB usan `where: { userId }`
- Todos los `db.*.create()` incluyen `userId` en el payload
- `resetMemoryState()` → `resetMemoryState(userId)`

### Migración DB
- `prisma/migrations/20260324000000_add_userid_isolation/migration.sql`
